# Issue #266 — Replace Logger message assertions with behaviour-focused expectations

**GitHub:** https://github.com/darthjee/navi/issues/266
**Label:** enhancement
**Impact:** Medium | **Effort:** Low

## Problem

Several tests assert on the exact string passed to `Logger.info` or `Logger.error`:

```js
expect(Logger.info).toHaveBeenCalledWith(`ResourceRequestJob #${job.id} performing`);
expect(Logger.error).toHaveBeenCalledWith(`Error occurred while performing job: #${job.id} - ${expectedError}`);
```

These assertions couple tests to logging implementation details. Renaming, rephrasing, or restructuring a log message breaks the test even if the underlying behaviour is correct, making refactoring unnecessarily painful.

## Affected files

- `source/spec/lib/models/ResourceRequestJob_spec.js`
- `source/spec/lib/models/Worker_spec.js`
- `source/spec/lib/services/Client_spec.js`

## Proposed change

Replace exact-message assertions with either:
- A call-count check (`toHaveBeenCalled()`) when the important fact is that logging occurred.
- A partial-match check (`toHaveBeenCalledWith(jasmine.stringContaining(...))`) when the important fact is that a key identifier (e.g. job ID) appears in the log.

Only keep full message assertions in `Logger_spec.js` and `BaseLogger_spec.js`, where the exact format is the subject under test.

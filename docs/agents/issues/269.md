# Issue #269 — Create shared test factories for common domain objects (Job, Resource, Client)

**GitHub:** https://github.com/darthjee/navi/issues/269
**Label:** enhancement
**Impact:** High | **Effort:** Medium

## Problem

Throughout the spec suite, common domain objects such as jobs, resources, and clients are built inline in each test file, often with repeated boilerplate:

```js
JobFactory.build('ResourceRequestJob', { attributes: { clients } });
```

There is no single, canonical factory module for these objects. Different test files use slightly different attribute sets, making tests inconsistent and fragile when the object's constructor signature changes.

## Affected files

Multiple spec files under `source/spec/lib/` including:
- `source/spec/lib/registry/JobRegistry_*_spec.js`
- `source/spec/lib/models/Worker_spec.js`
- `source/spec/lib/models/ResourceRequestJob_spec.js`
- `source/spec/lib/services/Engine_spec.js`

## Proposed change

Create a `spec/support/factories/` directory with factory modules for the most-used domain objects:
- `JobFactory.js` — builds `ResourceRequestJob` and other job types with sensible defaults and override support.
- `ResourceFactory.js` — builds `Resource` instances.
- `ClientFactory.js` — builds `Client` instances.

Each factory exposes a `build(overrides)` function. Replace all inline object construction across the spec suite with calls to the appropriate factory.
